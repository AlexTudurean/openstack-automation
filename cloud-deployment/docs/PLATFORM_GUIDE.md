# Platform Guide

> Scope: **cloud deployment / infrastructure**. Portal app guide: see `portal/docs/PLATFORM_GUIDE.md`.

This document explains how to access and operate the private cloud — dashboards, CLIs, health checks, network topology, and where to find passwords.

---

## Architecture Overview

```
Your Mac (192.168.0.x)
    │
    └── Proxmox VE   (192.168.0.100:8006)
            ├── controller  (192.168.0.101) — OpenStack control plane, HAProxy, keepalived
            ├── compute     (192.168.0.102) — Nova compute, Neutron OVN agent
            ├── storage     (192.168.0.103) — K3S control-plane, Rook Ceph OSDs
            └── portal      (192.168.0.104) — PaaS portal (outside OpenStack)

OpenStack internal network (VLAN 4000):  10.0.1.0/24
OpenStack external network (vmbr1):      10.0.2.0/24   ← floating IPs, Proxmox gateway 10.0.2.1
OpenStack tenant network (demo-net):     10.0.0.0/24   ← dev VMs
OpenStack mgmt network (mgmt-net):       10.0.4.0/24   ← dev VM admin NICs
VPN tunnel (admin):                      10.99.0.0/24
VPN tunnel (user):                       10.99.1.0/24
OpenStack API VIP (HAProxy/keepalived):  10.0.1.254
```

Two VPN profiles exist:

| VPN | Port | Tunnel | Access | Used by |
|---|---|---|---|---|
| **Admin** | 51820 | 10.99.0.x | Everything: 10.0.1.x, 10.0.2.x, 10.0.4.x, 192.168.0.x | Sysadmins |
| **User** | 51821 | 10.99.1.x | Tenant VMs only: 10.0.0.0/24 | Employees |

## 0. Admin WireGuard — One-time Setup

WireGuard runs on the **Proxmox host** (192.168.0.100). Always up regardless of VM state.

### Step 1 — Install and configure on Proxmox

```bash
ssh root@192.168.0.100
apt install -y wireguard

wg genkey | tee /etc/wireguard/server.key | wg pubkey > /etc/wireguard/server.pub
chmod 600 /etc/wireguard/server.key
wg genkey | tee /etc/wireguard/mac.key | wg pubkey > /etc/wireguard/mac.pub
```

Create `/etc/wireguard/wg0.conf`:
```ini
[Interface]
Address    = 10.99.0.1/24
ListenPort = 51820
PrivateKey = <contents of /etc/wireguard/server.key>

PostUp   = iptables -A FORWARD -i wg0 -j ACCEPT; \
           iptables -A FORWARD -o wg0 -j ACCEPT; \
           iptables -t nat -A POSTROUTING -s 10.99.0.0/24 -o vmbr0 -j MASQUERADE; \
           ip route add 10.0.1.0/24 via 192.168.0.101 dev vmbr0
PostDown = iptables -D FORWARD -i wg0 -j ACCEPT; \
           iptables -D FORWARD -o wg0 -j ACCEPT; \
           iptables -t nat -D POSTROUTING -s 10.99.0.0/24 -o vmbr0 -j MASQUERADE; \
           ip route del 10.0.1.0/24 via 192.168.0.101 dev vmbr0

[Peer]
PublicKey  = <contents of /etc/wireguard/mac.pub>
AllowedIPs = 10.99.0.2/32
```

```bash
echo "net.ipv4.ip_forward=1" >> /etc/sysctl.conf && sysctl -p
systemctl enable --now wg-quick@wg0
```

### Step 2 — Mac client config

```ini
[Interface]
PrivateKey = <contents of /etc/wireguard/mac.key from Proxmox>
Address    = 10.99.0.2/32

[Peer]
PublicKey           = <contents of /etc/wireguard/server.pub from Proxmox>
Endpoint            = 192.168.0.100:51820
AllowedIPs          = 10.0.1.0/24, 10.0.2.0/24, 10.0.4.0/24, 192.168.0.0/24, 10.99.0.0/24
PersistentKeepalive = 25
```

> **`10.0.4.0/24` must be in AllowedIPs** — this is the mgmt-net for direct admin SSH into dev VMs. Proxmox has a static route for it (`post-up ip route add 10.0.4.0/24 via 10.0.2.200` on vmbr1 in `/etc/network/interfaces`), so packets for 10.0.4.x go: Mac → Proxmox → OVN mgmt-router (10.0.2.200) → dev VM mgmt NIC.

### Step 3 — Resources accessible with admin VPN active

| Resource | Address |
|---|---|
| Proxmox web UI | `https://192.168.0.100:8006` |
| OpenStack Horizon | `http://10.0.1.254:8080` |
| Ceph dashboard | `http://ceph.tudu.io:32680` (add `10.0.1.4 ceph.tudu.io` to `/etc/hosts`) |
| Portal | `https://cloud.tudu.io` (add `192.168.0.104 cloud.tudu.io` to `/etc/hosts`) |
| Dev VM admin SSH | `ssh ubuntu@10.0.4.x` (mgmt-net, direct) |

## 0.5 User VPN — Employee Access

Employees connect to the **user WireGuard VPN** to reach their dev servers.

```
Employee Mac
    └── WireGuard (port 51821, tunnel 10.99.1.x)
            └── vpn-gateway VM (floating 10.0.2.150)
                    └── demo-net (10.0.0.0/24) — dev VMs
```

The vpn-gateway runs WireGuard (server) and dnsmasq. This is provisioned automatically in Phase 6. Employee VPN config is generated by the portal when a user account is created — employees download it once from the portal UI.

Employee `AllowedIPs`: `10.0.0.0/24, 10.99.1.0/24` — tenant VMs and tunnel only. Management VLAN, services VLAN, and OpenStack APIs are unreachable via user VPN by design.

`10.99.1.1` belongs to the vpn-gateway and is also the DNS server advertised in client configs. Portal users start at `10.99.1.2`. When a dev server is reserved, the portal updates `/etc/hosts` on the vpn-gateway and reloads dnsmasq so clients can SSH by short hostname:

```bash
ssh ubuntu@devserver-medium-001
ssh ubuntu@devserver-medium-001.cloud.internal
```

When the server is released, the portal removes those DNS entries, restores `pool-sg`, rebuilds the VM, and the background watcher returns it to the available pool after Nova reports ACTIVE.

The user VPN is routed, not NATed. The vpn-gateway forwards `10.99.1.0/24` traffic to demo-net while preserving the client's tunnel source IP. `demo-router` has a static return route:

```text
10.99.1.0/24 via 10.0.0.41
```

This is required because per-user security groups allow SSH from the user's tunnel IP, for example `10.99.1.2/32`. If the vpn-gateway MASQUERADEs user traffic, dev VMs see `10.0.0.41` instead, Neutron drops the packet, and SSH appears to hang.

### Manual: add a user peer to vpn-gateway

```bash
# From the portal VM:
ssh -i /opt/portal/keys/vpn_gateway.key ubuntu@10.0.2.150

# Or from the controller:
sudo ssh -i /root/.ssh/id_ecdsa ubuntu@10.0.2.150

wg genkey | tee /tmp/user.key | wg pubkey > /tmp/user.pub
sudo wg set wg0 peer $(cat /tmp/user.pub) allowed-ips 10.99.1.X/32
sudo tee -a /etc/wireguard/wg0.conf << EOF

[Peer]
PublicKey  = $(cat /tmp/user.pub)
AllowedIPs = 10.99.1.X/32
EOF
```

Client config:
```ini
[Interface]
Address    = 10.99.1.X/32
PrivateKey = <from /tmp/user.key>
DNS        = 10.99.1.1

[Peer]
PublicKey           = <server pubkey — cat /etc/wireguard/wg0.conf | grep PublicKey>
Endpoint            = <proxmox-public-ip>:51821
AllowedIPs          = 10.0.0.0/24, 10.99.1.0/24
PersistentKeepalive = 25
```

> The portal automates this entirely — manual steps only needed for testing without the portal.

## 4. OpenStack

All API endpoints and Horizon are behind HAProxy at VIP `10.0.1.254`. With admin VPN active, access Horizon directly at `http://10.0.1.254:8080`.

**Credentials:**
```bash
ssh sysoperator@192.168.0.101
sudo grep OS_PASSWORD /etc/kolla/admin-openrc.sh
```

**OpenStack CLI:**
```bash
ssh sysoperator@192.168.0.101
source /etc/kolla/admin-openrc.sh
export PATH=/opt/venv/kolla/bin:$PATH

openstack network agent list
```

| Network | CIDR | Purpose |
|---|---|---|
| `public1` | 10.0.2.0/24 | External (floating IPs) — gateway Proxmox 10.0.2.1 |
| `demo-net` | 10.0.0.0/24 | Tenant — dev VMs primary NIC |

## 5. PaaS Portal

**URL:** `https://cloud.tudu.io` (no VPN needed — portal is on home LAN)
**Mac `/etc/hosts`:** `192.168.0.104 cloud.tudu.io`

The portal runs as Docker Compose on the portal Proxmox VM (192.168.0.104):
- **nginx** — TLS termination, serves React SPA, proxies `/api/`
- **fastapi** — reserve/release/resize/migrate VMs, manage WireGuard peers
- **postgres** — users, dev VM pool state, reservations

### Portal VM networking

| Interface | Address | Reaches |
|---|---|---|
| `eth0` | 192.168.0.104/24 | Home LAN, internet |
| `eth0.4000` | 10.0.1.200/24 | OpenStack API (10.0.1.254) |
| Route via 192.168.0.100 | — | 10.0.2.0/24 (floating IPs, vpn-gateway) |
| Route via 192.168.0.100 | — | 10.0.4.0/24 (mgmt-net, dev VM SSH) |

### Bootstrap first admin user (one-time after Phase 7)

```bash
ssh sysoperator@192.168.0.104
docker exec -it portal-backend-1 python -m app.scripts.create_admin
```

The script prompts for email and password, generates a WireGuard keypair, and inserts the admin row with `is_admin=True`. `10.99.1.1` is reserved for the vpn-gateway/DNS, so the first user should use `10.99.1.2`. Run only once — subsequent users are created via the portal UI.

### Deploy / redeploy portal

```bash
cd cloud-deployment
./deploy.sh --phase 7
```

Or just:
```bash
ansible-playbook -i inventory/inventory.ini --vault-password-file .vault_pass deploy-portal.yml
```

### Portal health check

```bash
ssh sysoperator@192.168.0.104
cd /opt/portal && docker compose ps
docker compose logs --tail=50 backend
```

## 7. Running a Full Redeploy

### Software-only teardown (VMs stay up)

```bash
cd cloud-deployment
./teardown.sh      # kolla destroy → K3S reset → wipe /dev/sdb → clean config
./deploy.sh        # full redeploy phases 1-7
```

### Full reset (destroy and recreate Proxmox VMs)

```bash
# From Mac:
./teardown.sh

# On Proxmox (SSH as root):
bash lab/proxmox-teardown.sh   # destroys VMs 100-103, removes vmbr1
scp ~/.ssh/openstack_key.pub root@192.168.0.100:/root/openstack_key.pub
bash lab/proxmox-setup.sh      # recreates everything

# From Mac:
./deploy.sh
```

> `proxmox-setup.sh` creates VM 103 (portal) automatically and handles all iptables rules and the mgmt-net static route (via `post-up` on vmbr1). The only manual post-setup step is updating the admin WireGuard `AllowedIPs` on the Mac to include `10.0.4.0/24` if not already done.

### Applying OpenStack config changes to a running deployment

```bash
# Push updated globals.yml + restart affected containers:
ansible-playbook -i inventory/inventory.ini --vault-password-file .vault_pass \
  deploy-openstack.yml --tags configure_kolla,kolla_reconfigure
```
